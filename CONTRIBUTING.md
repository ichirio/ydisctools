# Contributing to ydisctools

`ydisctools` is a **Proof-of-Concept grab-bag** of small CDISC tools (SDTM/ADaM
builders, Define-XML / Pinnacle 21 readers, ISO 8601 helpers, CT lookups,
byte-aware text splitting, summary-table shaping, and the `plot_sankey()`
treatment-flow plotter). It is the workshop that
[rtfreporter](https://github.com/ichirio/rtfreporter) was spun out of. **No CRAN
submission is planned and the API may move.**

These rules apply to **every contributor — human or AI agent.** They are modelled
on rtfreporter's workflow, adapted for this PoC repo. If something here is
genuinely undecided, fall back to rtfreporter's conventions and adapt sensibly.

---

## Workflow: issue → branch → PR → green CI → merge

Any change to package code, docs, or tests goes through a pull request:

1. **`gh issue create`** — title + body in **English**. For a non-trivial change,
   put a **file checklist** in the issue body (everything you expect to touch:
   `R/…`, `man/…`, `NAMESPACE`, `DESCRIPTION`, `_pkgdown.yml`, tests, `NEWS.md`)
   so progress is visible.
2. **Branch off the latest `main`:** `git checkout -b <type>/<issue>-<slug>`
   (e.g. `feat/1-stat-table-helpers`, `fix/12-...`, `docs/…`). Types:
   `feat` / `fix` / `docs` / `refactor` / `test` / `chore`.
3. **Implement + commit.** Reference the issue in the commit/PR (`Closes #N`).
   Keep the whole documentation surface in sync (see below).
4. **`git push` + `gh pr create`** (English body).
5. **CI must be green** before merging — `R-CMD-check` (ubuntu-release) and
   `pkgdown`. Single maintainer ⇒ admin-merge is acceptable:
   `gh pr merge N --merge --admin --delete-branch`, then sync local main
   (`git fetch origin main && git reset --hard origin/main`).

- A plain question with **no** repo change needs no issue.
- Issues the maintainer directs in chat are **pre-approved** — execute them and
  add the `exec:agent` label for the record.
- Co-author trailer on commits:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.

### `exec:*` labels (dispatch axis, mirrored from rtfreporter)

- **`exec:agent`** — pre-approved; an agent works it.
- (No change / report-only / hold) — leave unlabelled and only report.

## ⚠️ Do NOT run `devtools::document()`

Most `man/*.Rd` are roxygen-generated, **but `man/plot_sankey_polygon.Rd` is
hand-maintained** (it carries `\name{plot_sankey}` while the roxygen in
`R/plot_sankey_polygon.R` uses `@rdname plot_sankey`). Running `document()`
generates a conflicting `plot_sankey.Rd` (duplicate `\alias{plot_sankey}`) and
**breaks R CMD check / pkgdown.**

So we maintain generated files **by hand:**

- Edit the **roxygen in `R/*.R`** (the source of truth / design doc) **and** the
  corresponding **`man/*.Rd`** so they stay in sync. The `.Rd` is what pkgdown
  renders.
- **`NAMESPACE` is hand-edited too** (despite its "Generated by roxygen2" header).
  Add `export(...)` / `importFrom(...)` lines yourself.

When adding a new exported function: write `R/<fn>.R` with roxygen, hand-write
`man/<fn>.Rd`, add `export(<fn>)` to `NAMESPACE`, and list the topic under the
appropriate `reference:` group in `_pkgdown.yml`.

## Keep the documentation surface in sync

On any feature/behaviour change, update **all** of: the function code + its
roxygen, the hand-written `man/*.Rd`, `_pkgdown.yml` (`reference:` index for new
topics), tests, and `NEWS.md` (the single changelog).

## Dependencies

If you add `@import` / `@importFrom` for a new package, add it to **both**
`NAMESPACE` **and** the `DESCRIPTION` `Imports:` list. A NAMESPACE/DESCRIPTION
mismatch makes clean installs (CI) fail with "no package called '…'".

## Versioning

- PoC package, **no CRAN submission**; the `NEWS.md` tracks notable changes only.
- Each development PR bumps the **PATCH only** (e.g. `0.0.228` → `0.0.229`). Pure
  doc/typo fixes may skip the bump.
- **MINOR/MAJOR bumps only on the maintainer's explicit instruction.** Never
  raise minor "because it's a new feature".
- `DESCRIPTION` `Version:` is digits-and-dots only.

## Tests

- testthat **edition 3** (`Config/testthat/edition: 3`). One `test-<topic>.R`
  per source file under `tests/testthat/`.
- Follow the existing naming convention: `test_that("<fn> Test N: <what>", …)`
  with a `## <fn> Test N: … ----` section comment.

---

## GitHub Pages / CI (do not "fix" the deploy mechanism)

- The site is published to <https://ichirio.github.io/ydisctools/>.
- **Pages source is `build_type: workflow`** (GitHub Actions artifact). The
  `pkgdown.yaml` workflow builds the site and deploys via
  `actions/configure-pages` + `upload-pages-artifact` + `actions/deploy-pages`.
  It does **NOT** push a `gh-pages` branch. Do **not** switch it to
  peaceiris/`gh-pages` — that would silently stop publishing unless the repo
  Pages source is also changed. (rtfreporter uses gh-pages; ydisctools does not.)
- `/docs/` is gitignored (local pkgdown build output).
- Verify the site builds locally before pushing site changes:
  `pkgdown::check_pkgdown(".")`.

## Environment notes

- On Windows/PowerShell the shell cwd resets to the home directory after each
  command — `Set-Location C:\Yrepo\Rrepo\ydisctools` (or `cd` in bash) first.
- Prefer ASCII-only source; build non-ASCII literals with `\uXXXX` / `intToUtf8()`.
- Some local R builds have unreliable `gsub` back-references — parse with
  `regmatches`/`gregexpr(perl = TRUE)` capture groups instead.
